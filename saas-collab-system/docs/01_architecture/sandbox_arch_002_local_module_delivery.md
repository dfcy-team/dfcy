# SANDBOX-ARCH-002 模块化本地 Sandbox 与精简交付合同

## 1. 决策

当前项目采用以下交付链路：

```text
模块合同简表
  -> 开发人员本地 Sandbox
  -> 模块 PR 与自动 CI
  -> 独立审核
  -> main 不可变制品
  -> 生产发布门禁
  -> 冒烟验证与回滚观察
```

本合同取代 `SANDBOX-ARCH-001` 作为当前默认交付模式。`SANDBOX-ARCH-001` 保留为独立基础设施 Sandbox 的严格参考，不再作为当前两台生产虚拟机之外必须常驻第三套环境的要求。

本次精简降低重复文档、重复复审和环境数量，不降低 tenant、权限、密钥、迁移、不可变制品、回滚和高风险动作边界。

## 2. 环境定义

| 环境 | 用途 | 允许内容 | 禁止内容 |
|---|---|---|---|
| Local Sandbox | 模块开发、Mock、集成和回归 | 合成数据、本地 MySQL/Redis、Mock、dry-run | 生产数据、生产凭据、真实平台动作 |
| GitHub CI | 全量测试、安全扫描、不可变制品生成 | 固定 Git SHA、镜像 digest、SBOM、测试证据 | 真实平台密钥、生产数据库连接 |
| Production | 经审批的正式运行环境 | 仅部署批准的同一镜像 digest | 目标主机构建、开发分支部署、未审批迁移 |

生产应用主机和数据库主机不得同时承载开发人员 Sandbox。开发人员 Sandbox 不构成生产授权，也不允许真实平台或资金操作。

## 3. 模块责任

每个模块必须登记负责人、目录、API、模型、迁移、权限、data_scope、状态机、上游、下游和测试。责任边界是代码所有权，不是绕过公共合同的授权。

示例分工：

| 模块 | 负责人 | 主要职责 | 禁止直接修改 |
|---|---|---|---|
| 销售库存财务对账 | 开发A | 销售指标、库存、预警、补货建议输入、对账建议与差异处理 | 达人状态、采购审批、付款、转账、提现及绕过财务独立授权的最终确认 |
| 达人管理 | 开发B | 达人档案、合作任务、内容和效果统计 | 库存主数据、采购状态、财务数据 |
| 供应链采购 | 架构员 | 采购订单、供应商任务、出货和审批合同 | 销售原始数据、达人业务、财务确认 |
| 公共底座 | 指定维护人 | tenant、账号、权限、审计和统一响应 | 业务模块不得单方面改变公共合同 |

架构员参与业务实现时不得自审、自批或单独合并该模块，必须由另一名审核人交叉复审。

## 4. Local Sandbox Profiles

标准 Sandbox 应提供以下逻辑 Profile：

- `core`：MySQL 8.4、Redis 7、Django、Celery、认证、tenant、权限和审计。
- `sales-inventory-finance-reconciliation`：销售、库存、财务对账模块及其合成和脱敏数据。
- `creator-management`：达人管理模块及其合成数据。
- `procurement`：供应链采购模块及其合成数据。
- `integration`：所有模块的本地前后端合同联调。
- `release-candidate`：只拉取 GitHub CI 生成的固定镜像摘要。

开发阶段允许本地构建；`release-candidate` 和 Production 禁止本地构建。未启动的模块由冻结合同 Mock 提供，Mock 不得标记为 `connected`。

## 5. 数据与集成边界

1. 本地可以共用一个 MySQL 容器，但各模块只写自己拥有的模型和表。
2. 跨模块写入必须通过服务层或冻结 API，禁止直接修改其他模块模型。
3. 所有核心资源必须有 tenant 边界；跨模块引用必须重新校验 tenant、权限、data_scope 和状态。
4. 仅允许合成数据。禁止复制生产订单、供应商、客户、达人、财务、银行或平台凭据。
5. 补货建议不能直接创建正式采购订单；预警不能触发真实 RPA；分析不能直接改变业务状态。
6. 财务保持独立权限域。开发A负责财务对账实现不代表销售库存权限自动获得财务能力；普通 internal、external 和 RPA 均不能借模块权限访问财务敏感数据。
7. 自动匹配只能产生对账建议；差异确认和最终对账必须校验独立 `finance.*` action permission，且不得执行付款、转账或提现。

## 6. 风险分级

### L1 低风险

适用于样式、文案、只读展示、Mock 和不改变合同的页面调整。

门禁：模块测试、PR、自动 CI、一名独立审核人。

### L2 中风险

适用于普通业务增删改、库存、采购、达人合作流程和非敏感导出。

门禁：合同简表、模块测试、tenant/权限测试、迁移检查、PR、自动 CI、一名独立审核人。

### L3 高风险

适用于 tenant、认证、权限、data_scope、财务、RPA、审批、状态机、审计、密钥、数据库迁移和真实平台边界。

门禁：完整合同、安全测试、全量 CI、架构复审、发布审批和回滚验证。发现 P1 后必须整改复审。

风险等级不得由代码作者单独降低。涉及多个等级时按最高等级执行。

## 7. 精简审核规则

1. 每个模块使用一份合同与验收简表，不再默认拆分多份任务、整改和收尾文档。
2. 无 P0/P1 时不固定执行 R2/R3；发现 P0/P1 后才进入定向整改和复审。
3. L1/L2 PR 运行变更模块测试；合并 `main` 或发布前运行全量测试。
4. 合同与实现可在一个 L1 PR 中完成；L2/L3 或跨模块变更必须先冻结合同。
5. 开发人员不能批准自己的 PR。架构员实现业务模块时同样适用。
6. 前端权限仅用于展示，后端仍是唯一可信授权边界。

## 8. 不可降低的安全底线

- 禁止提交或输出真实密码、Token、Cookie、Session、API Key、API Secret 和私钥。
- tenant、permission-specific data_scope、exact action permission 必须由后端执行。
- 财务必须独立授权、脱敏并保留审计。
- external 只能访问自己的数据；RPA Agent 只能访问 `/api/rpa/*`。
- 高风险状态变化必须由授权人员确认，禁止通用 PATCH、模型 save 或 bulk 绕过。
- 禁止开发电脑连接生产数据库或使用未经脱敏的生产数据。
- 禁止向生产主机复制工作目录并现场构建。
- Production 只接受受保护 `main` 的固定 Git SHA 和 CI 镜像 digest。
- 数据库迁移前必须备份、检查兼容性并准备回滚。
- 真实平台、真实 RPA、自动采购、清仓、改价和资金操作仍需专项评审。

## 9. 合同变更流程

跨模块 API、公共模型、权限、data_scope 或状态机变化必须：

1. 更新模块合同和接口映射。
2. 标明受影响模块及负责人。
3. 提供兼容或迁移方案。
4. 由至少一名非作者审核。
5. 在 CI 执行消费者和提供者合同测试。

不存在的接口必须标记 `pending` 或 `mock`；只有实际联调和 CI 证据通过后才能标记 `connected`。

## 10. 生效与退出

本合同合并 `main` 后生效。若后续新增独立 Staging/Sandbox 基础设施，可通过新的架构决策恢复严格多环境链路，但不得削弱本合同的不可变制品和生产安全底线。
