# UI-P7 治理与受控试点范围

## 1. 阶段目标

UI-P7 是用户界面分期计划的最后一期。目标是在 UI-P1 至 UI-P6 已完成的认证、权限、业务、RPA、审批、分析和 API 接入底座上，建设可审计的 API 治理、智能体占位、双主机试点就绪、备份恢复、发布回滚、容量观测和低风险灰度工作面。

本阶段只证明系统具备受控试点的治理与验证能力，不等于生产发布，也不授权真实平台、真实资金或高风险自动化。

## 3. 角色任务拆分

### 架构设计员

- UI-P7-ARCH-001 冻结页面、API、权限、data_scope、状态和审计合同。
- UI-P7-ARCH-002 审核双主机网络边界、密钥托管、备份恢复和回滚方案。
- UI-P7-ARCH-003 审核智能体占位的数据、工具、输出和人工确认边界。
- UI-P7-ARCH-004 审核低风险灰度门禁、停止条件和真实平台专项评审边界。
- UI-P7-ARCH-005 执行 G0 至 G5 门禁并输出最终准入结论。

### 开发A

- UI-P7-A-001 建设治理目录和版本/兼容性只读 API。
- UI-P7-A-002 建设试点就绪、拓扑、恢复计划、发布计划和容量观测模型及 API。
- UI-P7-A-003 落实 tenant/system scope、permission、data_scope、脱敏、幂等和不可变审计。
- UI-P7-A-004 只提供 Mock 检查、计划登记和结果回写，不在 Web 请求内执行主机命令、备份、恢复或部署。
- UI-P7-A-005 补齐 Django、迁移、权限、恢复/回滚状态机、数据质量和安全测试。

### 开发B

- UI-P7-B-001 建设 API 治理目录和合同详情。
- UI-P7-B-002 建设智能体治理只读占位，不提供真实模型凭据或外部工具连接。
- UI-P7-B-003 建设试点就绪、拓扑、备份恢复、灰度回滚和容量观测页面。
- UI-P7-B-004 覆盖 loading、empty、error、401、403、404、409、422、degraded、offline 和 stale。
- UI-P7-B-005 显示门禁证据、阻断原因、审批状态和审计引用，不在前端执行可信判断。

### 测试人员

- UI-P7-QA-001 执行双主机、端口隔离、健康链路和最小权限验证。
- UI-P7-QA-002 执行备份恢复、回滚、容量和低风险灰度演练并保存脱敏证据。
- UI-P7-QA-003 执行 tenant、system scope、角色、data_scope、敏感字段和审计越权测试。
- UI-P7-QA-004 执行浏览器 E2E、后端全量测试、前端测试、构建、Docker 和安全扫描。

## 4. 治理边界

1. API 合同只有一个现行版本；兼容别名必须有废弃日期，不得长期维护同义重复路径。
2. `connected` 仅表示本项目受控环境中的后端能力通过验收，不表示外部平台生产连接。
3. 所有门禁证据包含 commit、环境、时间、执行人、结果和脱敏附件引用。
4. 前端不得修改 CI、安全扫描、备份、恢复或回滚的原始证据。
5. 试点状态由后端门禁聚合计算；前端只显示结果和阻断项。
6. 名称包含 `mock` 仅代表计划实现固定示例行为；handler、测试和证据未完成前仍为 `pending（planned mock）`。
7. Mock实现通过测试后才可标记 `mock`；真实认证会话和完整验收通过后才可标记 `sandbox/connected`。

## 5. 智能体占位边界

- 仅登记 demo/placeholder 能力声明、允许数据分类、工具白名单、输出类型和人工复核要求。
- 不接入 OpenAI 或其他外部模型，不配置真实 API Key，不执行网络工具，不写业务数据库。
- Mock 评估只使用固定示例输入，输出不得触发采购、改价、清仓、上下架、RPA 或资金操作。
- 智能体建议必须标记来源、置信度、限制和人工确认状态；不得作为可信权限或业务结论。

## 6. 双主机与数据边界

- 数据库主机只运行 MySQL 8.4 和受控备份任务，不运行前端、RPA 或外部平台连接器。
- 应用主机运行 Nginx、Django、Celery 和 Redis；后端端口仅绑定回环地址，外部只经 Nginx 进入。
- MySQL、Redis、Celery broker 和管理端口不得暴露公网；拓扑页面不得显示完整 IP、密码或连接串。
- `.env.pilot`、备份密钥和数据库凭据只由受控主机或密钥托管系统提供，不进入 Git 或 API 响应。

## 7. 恢复、回滚与灰度边界

- 页面只能创建演练/发布计划、发起审批和记录外部受控执行结果，不直接运行 shell、Docker、SQL 恢复或部署命令。
- 恢复和发布只允许通过合同冻结的专用动作端点迁移状态；禁止通用状态 PATCH、直接 model save、QuerySet update 或后台绕过审批。
- 创建人与审批人必须分离；审批、记录和回滚使用独立 exact permission 及独立 data_scope。
- 发布记录人工恢复使用 `resume` 与 `pilot.release.record`；回滚人工恢复使用独立 `resume-rollback` 与 `pilot.release.rollback`，不得共享权限判断。
- 原发布审批不自动授权回滚；回滚必须另行批准并记录独立 approval ref，回滚批准人与结果记录人必须分离。
- 备份恢复必须记录 RPO、RTO、备份摘要、校验结果、审批人和审计引用；不得存储备份正文或密钥。
- 灰度仅限本项目低风险只读能力和 demo tenant；必须定义观察窗口、停止条件和回滚点。
- 回滚必须关联已验证 Git commit/tag、数据库迁移兼容性和备份点；禁止 `reset --hard` 或未经批准的数据回退。

## 8. 阶段不做事项

- 不接入真实 BigSeller、Shopee、TikTok/TK、银行、支付或真实 AI 模型服务。
- 不提交真实账号、密码、Token、Cookie、Session、API Key、API Secret、备份密钥或主机私钥。
- 不由页面直接执行部署、备份、恢复、回滚、主机命令或网络策略修改。
- 不自动采购、改库存、刊登、改价、清仓、停售、归档、付款、转账或提现。
- 不让预警、分析、智能体或治理检查触发真实 RPA。

## 9. 交付顺序

1. 冻结本范围、API/权限合同、总映射和验收清单。
2. 执行 `UI-P7-ARCH-CONTRACT-R1` 独立前审。
3. 前审无 P0/P1 后才允许实施后端模型、页面和 Mock/受控验证。
4. 完成自动化、浏览器、双主机、恢复、回滚、容量和安全验证。
5. 执行实现复审和发布审核；通过后才允许 UI-P7 正式收尾。
