# 阶段2范围说明

## 阶段2目标

阶段2在阶段1 MVP 的权限、tenant 隔离、RPA 协议和前后端联调基础上，完成可审计的 API 数据接入框架、商品状态识别框架、财务对账流程模型、供应商绩效框架、RPA 稳定性方案和 CI/CD 质量门禁。优先使用 Mock、脱敏样例和沙箱。

## 阶段2不做事项

- 不直接接入真实 BigSeller、Shopee、TK/TikTok、银行或支付平台。
- 不提交真实账号、密码、Token、Cookie、Session、API Key、API Secret 或证书。
- 不启用自动改价、清仓、补货、上架、下架、刊登或财务自动资金操作。
- 不让 RPA 直连 MySQL、Redis 或其他数据层；不让前端承担真实权限判断。

## 阶段2与阶段1边界

阶段1交付了 MVP 业务接口、权限与 tenant 基线、RPA 任务协议、Mock/API 切换和页面联调。阶段2只在这些边界之上扩展框架、模型、流程、沙箱和质量能力，不推翻既有接口分区：内部业务使用 `/api/internal/*`，供应商使用 `/api/external/*`，RPA Agent 使用 `/api/rpa/*`，财务使用 `/api/finance/*`。

## API数据接入范围

建立连接器契约、凭据托管要求、请求签名、限流、重试、幂等、webhook 验签、数据清洗、脱敏日志、质量校验、失败告警和灰度策略。阶段2仅验证 Mock 或经批准的沙箱，不连接真实平台。

## 自动商品状态识别范围

定义 API、RPA 页面回读和人工标记的状态来源、优先级、状态流转、自动建议与人工确认机制。自动识别只能给出建议，不能由 RPA 或规则直接作出清仓、停售等高风险业务决策。

## 财务对账增强范围

建立平台账单、取款记录、银行到账和手工导入的脱敏模型；定义自动匹配、差异标记、人工处理、财务独立授权和审计日志。阶段2不直连真实银行，不执行真实资金自动操作。

## 供应商绩效范围

以 `tenant_id + supplier_id` 为数据范围，定义交付及时率、逾期率、异常率、反馈及时性、出货准确率和供应商评分。供应商仅能查看自己的绩效；内部汇总查询须经后端权限控制。

## RPA稳定性增强范围

规划页面变化检测、失败截图、日志留存、任务幂等、重试上限、同账号串行、失败转人工和审批凭证校验。RPA 只回写执行结果，不作业务判断，且不得访问 finance、internal finance 或 admin。

## CI/CD完善范围

定义后端检查和测试、前端构建、契约测试、RPA JSON 校验、依赖扫描、密钥扫描和发布门禁。全部流水线使用非生产配置，不携带真实凭据或业务数据。

## 真实平台接入安全评审边界

每个真实平台必须单独完成凭据生命周期、最小网络权限、回调验签、数据分类、审计、限流、灰度、回退、人工接管和紧急停止评审，并取得书面批准后才能开始接入。平台审批不自动授权高风险 RPA 动作；该类动作仍需独立审批。
