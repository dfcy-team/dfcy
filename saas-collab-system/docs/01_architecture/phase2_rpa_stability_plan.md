# 阶段2 RPA 稳定性增强方案

## 稳定性能力

- 页面变化检测：检测页面关键状态或占位选择器失效，进入失败或人工接管，不静默继续。
- 失败截图：在关键失败节点保留脱敏截图元数据或占位，不提交真实账号、页面敏感信息或 Cookie。
- 日志留存：按 task_id 保存结构化步骤日志、错误码、重试次数和最后成功步骤，并进行敏感字段脱敏。
- 失败转人工：支持 `manual_required`、`manual_reason`、`failed_step` 和 `last_success_step`，由后端生成可处理任务。
- 重试上限：后端控制可重试错误和最大次数，超过上限转人工或取消。
- 同账号串行：同一逻辑账号范围的高风险任务必须串行，避免并发页面操作冲突。

## 审批与业务边界

高风险动作必须携带后端审批凭证；例如改价任务只能执行 `approval_status=approved` 的 approved_price。RPA 不做业务判断，不决定清仓、补货、上下架或刊登，不绕过后端直接写业务状态。

## 访问与平台边界

RPA Agent 只能访问 `/api/rpa/*`，不得访问 `/api/finance/*`、`/api/internal/finance/*` 或 `/admin/`，也不得直连 MySQL、Redis 或其他数据层。阶段2不得连接真实 BigSeller，除非真实平台专项安全评审已通过；即使通过平台接入评审，高风险自动化仍需单独批准。
