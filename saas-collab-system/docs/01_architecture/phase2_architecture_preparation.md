# 阶段2架构准备与边界说明

## 1. 目标与范围

阶段2在阶段1 MVP 通过后，优先完成 API 数据接入的安全方案、自动商品状态识别的规则框架、财务对账增强的设计、供应商绩效指标、RPA 稳定性方案和 CI/CD 准备。

本阶段准备工作只定义 Mock、沙箱、接口契约、模型演进方向、流程和验收标准。真实平台、真实凭据、真实银行或支付数据，以及真实高风险自动化均不在本准备范围内。

## 2. 不变系统边界

- API 只负责数据接入、验证、标准化和同步编排，不负责页面操作。
- RPA 只负责 BigSeller 页面操作和执行结果回写，不负责业务判断。
- RPA Agent 只能调用 `/api/rpa/*`，不得访问 `/api/finance/*`、`/api/internal/finance/*` 或 `/admin/`，也不得直连 MySQL、Redis 或其他数据层。
- 前端只负责展示、交互、表单和 API 调用；真实权限以后端返回的 roles、permissions、data_scope 为准。
- 供应商只能访问自身 tenant 范围内、且属于自身 supplier_id 的任务和绩效数据。
- 财务敏感接口必须经过独立财务权限码授权，普通 internal 用户不得默认取得访问权。
- MySQL 是最终可信业务数据存储；生产和 staging 不得使用 SQLite，MySQL 与 Redis 不得暴露公网。

## 4. 独立审批门

以下事项不得被普通开发任务隐含覆盖，必须单独立项、评审并记录结论：

1. 真实 BigSeller、Shopee、TK/TikTok、银行或支付平台接入。
2. 真实账号、密码、Token、Cookie、Session、API Key 的申请、托管、轮换和撤销。
3. 自动改价、清仓、补货、上架、下架和财务自动对账。
4. 生产网络连通、IP 白名单、回调地址和对象存储。
5. 涉及财务、供应商或跨 tenant 数据的导出与批处理。

未取得对应审批时，只允许使用 demo、placeholder、sandbox 或 Mock 数据。

## 5. 数据与权限原则

- 每个新增核心模型和查询必须体现 tenant 或 tenant_id；服务层、列表查询、详情查询和异步任务均须进行 tenant 过滤。
- 供应商数据必须叠加 `tenant_id + supplier_id` 过滤，不能由请求参数绕过。
- 财务数据采用最小权限原则，权限码至少预留 view、export、reconcile、import、exception.handle。
- 平台原始数据进入可信业务表前，应保留来源、同步批次、校验结果和审计关联信息。
- 日志和错误信息不得写入完整凭据、敏感财务字段或跨 tenant 数据。

## 6. 阶段2开发启动条件

每个工作包开始开发前必须具备：

1. 对应任务卡、责任人和验收标准。
2. API 契约或页面/任务状态说明，未完成接口标记为 mock 或 pending。
3. tenant、权限、审计和失败处理说明。
4. Mock 或沙箱数据来源，且不含真实业务数据或凭据。
5. 测试计划和回退策略。
