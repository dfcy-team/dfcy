# 阶段2商品状态识别状态机

## 商品状态来源

商品状态由三类来源构成：

- API 数据来源：平台同步的销量、库存、刊登和价格等经校验数据。
- RPA 回读来源：页面价格、刊登状态和页面快照等执行结果；RPA 只负责回读与回写。
- 人工标记来源：经授权内部人员对异常、清仓、停售或归档的确认。

## 状态优先级

1. 经授权的人工确认优先级最高。
2. 已批准的业务流程状态优先于自动识别建议。
3. 通过质量校验的 API 数据与 RPA 回读用于生成建议；数据冲突时进入待人工确认。
4. 未通过质量校验、过期或跨 tenant 数据不得推动状态流转。

## 状态定义与流转

| 状态 | 含义 | 允许的建议流转 | 确认边界 |
|---|---|---|---|
| 新品 | 已建档、处于导入或观察期 | 新品 -> 正常销售 / 停售 | 正常销售可由规则建议，停售需人工确认 |
| 正常销售 | 销售和刊登指标在规则范围内 | 正常销售 -> 滞销 / 清仓 / 停售 | 滞销可自动建议，清仓和停售必须人工确认 |
| 滞销 | 规则识别的低动销候选 | 滞销 -> 正常销售 / 清仓 / 停售 | 清仓、停售必须人工确认 |
| 清仓 | 已获审批的清库存业务状态 | 清仓 -> 停售 / 归档 | 必须有后端审批凭证 |
| 停售 | 停止销售或刊登的业务状态 | 停售 -> 归档 / 正常销售 | 恢复销售需人工确认 |
| 归档 | 历史保留，不再参与正常运营 | 无自动流转 | 只能由授权人工流程处理 |

## 自动建议与人工确认

自动状态识别只能生成候选状态、规则版本、证据摘要和建议原因。系统必须保留人工确认、驳回、调整和审计记录。自动建议不得直接触发改价、清仓、补货、上下架或刊登。

## RPA边界

RPA 不得自行决定商品业务状态，不得绕过后端写入状态，也不得以页面结果替代审批。RPA 回读仅作为经质量校验后的一个数据来源，且只能调用 `/api/rpa/*` 回写结果。
