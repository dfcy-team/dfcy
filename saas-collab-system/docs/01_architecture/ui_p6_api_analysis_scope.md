# UI-P6 API接入与分析复盘范围

## 1. 阶段目标

UI-P6 在 UI-P1 至 UI-P5 已完成的登录、权限、治理、RPA、审批和业务主链路基础上，统一经营看板、归因复盘、财务只读分析、生命周期复盘、清仓申请占位和 API 接入中心。所有指标、预警和建议只用于分析、解释和人工决策，不直接触发业务动作。

## 2. 页面范围

| 工作面 | 页面路径 | 主要任务 | 阶段状态 |
|---|---|---|---|
| 角色工作台 | `/` | 按后端权限展示入口、待办摘要和数据质量提示 | 保留并增强 |
| 经营总览 | `/analytics/overview` | 查看销售、库存、商品、供应商、财务和 RPA 的只读摘要 | 新增，pending |
| 销售归因复盘 | `/analytics/sales` | 按时间、平台、店铺、商品和国家钻取，只展示授权范围 | 增强，pending |
| 库存分析 | `/analytics/inventory` | 查看可售、在途、覆盖天数、缺货和超储风险 | 增强，pending |
| 财务只读复盘 | `/finance/analytics` | 查看脱敏后的账单、到账、差异和异常摘要 | 增强，pending |
| 生命周期复盘 | `/lifecycle/reviews`、`/lifecycle/history` | 查看建议、证据、置信度、人工结论和审计 | 增强，pending |
| 清仓申请 | `/lifecycle/clearance-requests` | 查看 `clearance` 审批；阶段内仅允许创建 Mock 申请 | 新增，mock |

页面不新增嵌套营销卡片或说明型落地页。列表、详情、筛选、指标摘要、数据质量、异常和审计信息应采用紧凑工作台布局。

## 3. 页面状态合同

每个数据页面必须具备 `loading`、`empty`、`error`、`401`、`403`、`404`、`409`、`422`、`degraded` 和 `offline` 状态。分页页面还必须保留筛选条件、当前页和刷新入口。

接口能力状态只允许：

- `mock`：仅返回演示数据，不发送真实业务请求。
- `pending`：代码或接口存在，但尚无本阶段联调证据。
- `sandbox`：仅在受控沙箱验证，不连接生产平台。
- `connected`：使用真实登录会话与本项目后端完成验收，且有测试证据。
- `degraded`：后端失败后展示明确降级状态，不伪装为实时数据。
- `disabled`：因权限、安全评审或生产边界被禁用。

前端不得根据 HTTP 200、配置 `active` 或存在凭据指纹自行判定外部平台已连接。

## 4. 分析与归因边界

1. 归因分析只组合后端已按 tenant、permission 和 data_scope 过滤的数据。
2. 前端不得跨 tenant、跨店铺或跨权限拼接指标。
3. 指标必须展示时间范围、口径版本、数据更新时间、质量状态和缺失值说明。
4. 缺失数据使用 `N/A` 或明确空状态，不补造零值。
5. 归因结论属于分析建议，不自动生成采购单、改价任务、清仓动作或 RPA 任务。

## 5. 财务边界

财务页面仅允许 `internal + finance.view`。账号、流水标识和敏感字段必须由后端脱敏；前端不得恢复、缓存或导出明文。页面不提供付款、转账、提现、自动确认对账或银行连接操作。财务导出、导入、对账和异常处理分别使用独立权限，不得由 `finance.view` 代替。

## 6. 生命周期与清仓边界

API、RPA 和规则只能生成生命周期建议。普通确认使用 `products.lifecycle.confirm`；清仓、停售和归档还必须由后端校验 `products.lifecycle.high_risk_confirm`。UI-P6 的清仓申请只进入 `approval_type=clearance` 的审批视图，Mock 创建不改变商品状态，不改价、不下架、不触发采购或 RPA。

## 7. API接入边界

API 接入中心只展示凭据别名、掩码、指纹、版本、环境和审计结果。`production` 配置默认禁用，真实连接、真实凭据录入和生产校验均不属于 UI-P6。同步动作仅允许 `run-mock`；失败日志必须脱敏并展示数据质量检查结果。

## 8. 阶段不做事项

- 不接入真实 BigSeller、Shopee、TikTok/TK、银行或支付平台。
- 不展示、保存或提交真实 Token、Cookie、Session、API Key 或 API Secret。
- 不自动采购、补货、刊登、改价、清仓、停售、归档或修改库存。
- 不让预警、看板或清仓申请触发真实 RPA。
- 不在前端承担 tenant、data_scope、财务权限或高风险动作的最终判断。

## 9. 交付顺序

1. 冻结本范围、API/权限合同和验收清单。
2. 执行独立架构合同前审。
3. 实施页面与最小后端补充，不改变既有安全边界。
4. 完成前后端、权限、数据质量、窄屏和异常状态测试。
5. 执行 UI-P6 架构复审，通过后才允许提交和创建 PR。
