# 用户界面功能运用设计任务拆分

## 1. 架构设计员

- UI-ARCH-001 冻结角色、菜单、页面、API、权限和状态合同。
- UI-ARCH-002 审核前端与后端的 tenant、data_scope、字段脱敏和导出边界。
- UI-ARCH-003 审核 API、RPA、人工三种执行方式及失败回退。
- UI-ARCH-004 为 UI-P0 至 UI-P8 执行 G0 至 G5 阶段门禁。
- UI-ARCH-005 维护 P0/P1/P2 清单、责任人、验收标准和复审结论。
- UI-ARCH-006 审核真实平台、Webhook、机器身份、凭据引用和网络分区。
- UI-ARCH-007 审核测试证据，不使用开发变更日志代替实际验证结果。
- UI-ARCH-008 在每期 main 合并后给出下一期准入结论。

## 2. 开发A（后端）

- UI-A-001 输出 `/auth/login`、`/auth/refresh`、`/auth/me` 和退出策略的真实合同。
- UI-A-002 输出菜单、permission、字段权限和 data_scope 的可信响应。
- UI-A-003 补齐页面所需 API、分页、统一错误、状态冲突和业务校验。
- UI-A-004 保证所有核心查询 tenant 隔离，并覆盖 external/RPA/finance 越权测试。
- UI-A-005 落实审批、幂等、审计、导出和敏感字段脱敏。
- UI-A-006 维护 Mock/Sandbox 适配器，production adapter 默认拒绝执行。
- UI-A-007 修复 MySQL 不支持 RPA 条件唯一约束的问题，再放行并发 RPA。
- UI-A-008 提供迁移、Django check、pytest、数据质量和回滚证据。

## 3. 开发B（前端）

- UI-B-001 完成真实登录、Token刷新、`/auth/me`、退出和401回登录。
- UI-B-002 建设统一 MainLayout、角色菜单、面包屑、环境和能力状态。
- UI-B-003 建设统一页头、查询、列表、详情、时间线、审计和操作反馈组件。
- UI-B-004 建设 loading、empty、error、403、409、422、partial、offline 状态。
- UI-B-005 按权限码控制展示，但不把前端展示当作可信授权结果。
- UI-B-006 按最终 API 合同处理分页、错误、幂等和 Mock/API 状态。
- UI-B-007 为桌面和窄屏提供稳定布局，控制表格、筛选和详情的可用性。
- UI-B-008 补充组件、路由、请求、认证和关键页面测试以及构建证据。

## 4. 测试人员

- UI-QA-001 建立角色、tenant、data_scope、按钮、字段和流程权限用例。
- UI-QA-002 覆盖正常、空数据、慢响应、离线、部分成功和统一错误状态。
- UI-QA-003 覆盖登录、刷新、退出、过期、禁用账号和跨租户访问。
- UI-QA-004 覆盖桌面、窄屏、键盘操作、文本溢出和无障碍基础检查。
- UI-QA-005 覆盖 API/RPA/人工边界、高风险审批和外部回填待确认。
- UI-QA-006 保存命令、版本、commit、截图、日志和结果，不伪造未执行项。

## 5. 产品与业务审核人

- 确认角色工作台、菜单语言、字段口径和日常任务顺序。
- 确认主操作、异常处理、审批原因、人工接管和关闭条件。
- 确认指标数字与下钻列表筛选一致。
- 确认敏感字段、导出、外部供应商和财务岗位的最小可见范围。

## 6. 责任规则

每个任务必须包含责任人、依赖、分支、API合同、测试用例、证据位置和验收人。架构员只给出审核结论，不代替开发修复；测试失败不得通过调整页面文案伪装为成功。
