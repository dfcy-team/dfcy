# 用户界面功能运用设计分期执行计划

## 1. 计划依据

本计划依据《跨境电商SaaS协同系统_用户界面功能运用设计规划_V2.1_API安全架构融合优化审核预览》制定。当前系统已具备 Django REST Framework、Vue 3/Vite、MySQL、Redis/Celery、Nginx、Mock/API 切换和阶段 1 至阶段 3 的业务底座。

执行原则：先冻结合同，再建设统一底座，随后按风险从低到高接入业务。每一期均采用“设计 -> 架构审核 -> 实现 -> 测试 -> 复审 -> 放行”的闭环，不允许跨过阶段门禁直接接入真实平台或高风险自动化。

## 2. 全局边界

- 前端负责展示、交互、表单和 API 调用，不执行可信权限判断。
- 后端负责 tenant、角色、permission、data_scope、字段脱敏、流程状态和高风险动作校验。
- API 负责数据接入，RPA 负责受控页面操作，二者状态和失败回退不得混用。
- MySQL 是最终可信业务数据存储；飞书、微信、RPA 和外部平台均不是主数据源。
- Mock、pending、sandbox、connected、degraded、disabled 必须基于证据显示。
- 真实发布、改价、改库存、采购、清仓、停售、归档、付款、转账和提现不在默认开发权限内。

## 3. 分期总览

| 阶段 | 目标 | 主要交付 | 必测范围 | 架构门禁 |
|---|---|---|---|---|
| UI-P0 | 现状审核与合同冻结 | 页面/路由/API/权限/RPA矩阵，统一UI合同，问题清单 | 静态扫描、现有测试、构建、部署健康 | P0/P1关闭后才进入P1 |
| UI-P1 | 统一界面底座 | 真实登录会话、角色菜单、页面骨架、状态组件、角色工作台 | 登录刷新、403/409/422、窄屏、组件测试、构建 | 前端不得绕过后端权限 |
| UI-P2 | 系统管理与基础档案 | 组织用户、六层权限、平台/店铺/仓库/供应商档案、安全运维页 | tenant隔离、权限矩阵、脱敏、审计 | 业务模块不得重复维护档案 |
| UI-P3 | RPA任务与设备基础 | 双状态机、任务/运行、设备、人工队列、账号锁与页面签名 | claim并发、心跳、超时、撤销、越权、Mock Worker | 仅Mock/dryRun设备可执行 |
| UI-P4 | 审批、报告、异常与协同 | 审批、报告、异常闭环、飞书/微信Mock、回访 | 审批绕过、Webhook验签/重放、断网恢复、下载审计 | 外部回填不得直接生效 |
| UI-P5 | 业务主链路接入 | 商品、采购、库存、刊登、订单、价格、供应商协同 | 主链路、幂等、待确认、data_scope、异常闭环 | 不自动采购、发布、改价或改库存 |
| UI-P6 | API接入与分析复盘 | 归因、财务只读、生命周期、清仓申请、API接入中心、看板 | 数据质量、映射、脱敏、只读、人工确认 | 指标和建议不得直接触发动作 |
| UI-P7 | 治理与受控试点 | API治理、智能体占位、双主机验证、恢复/回滚、低风险灰度 | E2E、安全扫描、网络隔离、备份恢复、容量 | 真实平台仍需专项评审 |
| UI-P8 | 生产试点运维与专项安全准入 | 试点控制台、安全评审、验证与性能证据、人工Go/No-Go | 权限分离、证据完整性、失效、E2E、容量与安全门禁 | 只允许形成准入结论，不执行部署或真实平台动作 |

## 4. 每期标准工作流

1. 产品/业务负责人确认用户角色、任务目标、输入输出和异常路径。
2. UI/前端输出信息架构、低保真流程、字段表、状态表和交互原型。
3. 后端负责人核对 API、权限、tenant、data_scope、状态机和审计落点。
4. 架构员执行设计前审，结论仅允许 PASS、CONDITIONAL_PASS、FAIL。
5. 开发在独立功能分支实现，不在架构审核分支修改业务代码。
6. 测试执行自动化、浏览器、权限、安全、构建和部署验证并保存证据。
7. 架构员执行实现复审；P0/P1 未关闭不得合并或进入下一期。
8. 合并 main 后执行试点回归并更新发布说明，不以标签代替 main。

## 5. 架构审核节点

- G0 范围审核：确认本期做什么、不做什么以及真实平台边界。
- G1 合同审核：冻结页面、API、权限码、字段、状态、错误码和审计要求。
- G2 设计审核：检查岗位适配、主链路、异常路径、敏感信息和高风险按钮。
- G3 实现审核：检查代码范围、前后端一致性、tenant/data_scope 和安全边界。
- G4 测试审核：核对实际命令、结果、失败项、截图、日志和测试数据来源。
- G5 发布审核：确认 main、部署包、回滚点、配置模板和准入结论。

## 6. 近期执行顺序

UI-P0 至 UI-P7 已按门禁完成并合入 `main`。UI-P8 作为原七期计划之后的受控扩展，只处理生产试点前的证据、专项安全评审、验证结果和人工准入决策；不得把“试点准备”解释为真实平台连接、生产部署或高风险自动化放行。
