# 用户界面设计阶段进入说明

## 当前结论

UI-P0结论为CONDITIONAL_PASS。允许进入用户界面信息架构、交互流程、低保真原型、公共组件和Mock设计；不代表真实业务登录、权限或生产能力已经验收。

## 首批工作

1. 真实登录、Token刷新、`/auth/me`和退出。
2. 角色化菜单、面包屑和页面访问状态。
3. 统一页头、查询、列表、详情、时间线和审计区。
4. loading、empty、error、403、409、422、partial、offline状态。
5. 系统管理员、管理层、运营、采购、财务和技术/RPA工作台低保真设计。

## 启动条件

- 开发分支必须基于最新main。
- UI/API/权限/状态合同必须经架构员确认。
- 每项任务必须关联测试用例和验收证据。
- Mock和真实API状态必须可见且准确。

## 禁止事项

- 不直接接入真实平台、真实飞书、真实微信、银行或支付。
- 不启用自动采购、改价、改库存、清仓、停售、归档和资金操作。
- 不以静态菜单隐藏代替后端权限。
- 不在MySQL账号锁整改前启动真实并发RPA。

## 后续门禁

UI-P1实现完成后，由架构员审核认证、角色菜单、统一状态和页面骨架；测试人员执行前端测试、构建、浏览器场景、后端权限和越权验证。无P0/P1后才允许进入UI-P2。
