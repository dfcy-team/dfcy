# 阶段2准备与发布门禁

## 1. 准备结论

阶段2可进入规划和 Mock/沙箱准备。阶段1 `v0.2-phase1-complete` 是当前基线，不等同于真实平台接入授权。

## 2. 开发启动门禁

1. 分支必须从最新 `main` 创建，并关联阶段2任务编号。
2. 每项变更必须限定在已批准的任务范围，禁止将真实平台配置、凭据或数据混入提交。
3. API、模型、任务和页面在实现前需明确 tenant、权限、审计、失败处理和 Mock/沙箱策略。
4. 涉及财务、供应商数据范围、RPA 高风险动作或外部平台的事项必须先通过专项评审。

## 3. 发布与演示门禁

- 后端检查和测试、前端构建、RPA JSON 校验、依赖扫描和密钥扫描须有可复现记录。
- 仅可使用开发或沙箱环境，MySQL/Redis 不得暴露公网，生产配置禁止 SQLite。
- 未完成接口必须标记为 mock 或 pending，不得伪造 connected。
- 任何真实平台、银行、支付、BigSeller、Shopee 或 TK/TikTok 接入均需独立安全评审、变更审批和回退方案。

## 4. 高风险自动化门禁

自动改价、清仓、补货、上下架、刊登和财务自动对账不得因阶段2任务自动获得执行权。后续若评审通过，也必须具备后端审批凭证、最小权限、幂等控制、审计日志、人工接管和紧急停止机制。
