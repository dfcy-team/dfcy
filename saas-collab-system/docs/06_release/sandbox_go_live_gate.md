# SANDBOX-ARCH-001 上线准入门禁

## 1. 适用范围

本门禁控制应用制品从 CI 进入 Sandbox、从 Sandbox 进入受控 Pilot，以及后续申请 Production 的流程。“上线”包括系统发布和任何可能影响真实商品、订单、供应商、财务、平台或 RPA 的能力启用。

## 2. 晋级路径

| 门禁 | 输入 | 允许输出 |
|---|---|---|
| G0 开发完成 | 已审核代码、测试和迁移 | 合并受保护 `main` |
| G1 CI 制品 | 固定 Git SHA、全部必需 CI 成功 | 可部署 Sandbox 的候选制品 |
| G2 Sandbox 准入 | 配置、安全开关、测试数据和变更单获批 | 允许部署独立 Sandbox |
| G3 Sandbox 验收 | 验收清单完成、P0/P1 为零、证据完整 | 允许申请受控 Pilot |
| G4 Pilot 准入 | 同一制品摘要、Pilot 风险评审和回滚方案 | 允许受控内网小范围试点 |
| G5 Production 评审 | 生产安全、容量、运维、凭据和业务审批 | 另行决定；当前默认禁止 |

任何门禁失败都必须停止晋级。禁止跳级、口头放行或用 Draft/未合并分支部署正式环境。

## 3. G1 CI 制品门禁

必须记录：

- Git SHA 与来源 PR。
- 后端、前端镜像不可变摘要。
- 数据库迁移清单。
- 后端和前端测试结果。
- Docker/RPA/文档检查结果。
- 依赖和密钥扫描结果。
- 制品 SBOM 或锁定依赖清单。
- OCI revision 必须等于固定 Git SHA；制品必须来自仓库 `Sandbox immutable artifacts` 工作流。
- 受控制品清单的环境、Git SHA、前后端摘要、迁移摘要及清单哈希必须与部署证据一致。

CI 失败、HEAD 变化、存在未审核提交或制品无法追溯时不得进入 Sandbox。

## 4. G2 Sandbox 部署门禁

1. 环境编码必须为 `sandbox`。
2. 网络、数据库、Redis、TLS、账号、密钥、日志和备份与其他环境独立。
3. 真实平台和高风险自动化开关必须关闭。
4. 测试数据必须为合成或脱敏数据。
5. 变更单必须包含部署步骤、验证负责人、失败停止条件和回滚步骤。
6. 部署前备份现有 Sandbox 数据并验证可恢复性。
7. 应用和数据库网络策略已经审批，`DOCKER-USER` 规则持久化且禁止未批准出站/来源；数据库正向连通、公共出站阻断、第三台未批准来源拒绝以及两台主机重启后规则持久化均有 mode `0400` JSON 实测证据。
8. 数据库中的 `sandbox` 环境资源为唯一、受控状态，并准备短期 JWT 验证账号。

## 5. G3 Sandbox 验收门禁

`docs/05_test/sandbox_acceptance_checklist.md` 中的阻断项必须全部完成。验收结果按以下规则判断：

- 存在 P0：`FAIL`，立即停止并评估环境隔离或数据泄露。
- 无 P0 但存在 P1：`CONDITIONAL_PASS`，不得晋级。
- 无 P0/P1：`PASS`，可提交 Pilot 准入申请。

验收报告必须固定 Git SHA 和镜像摘要。代码、依赖、迁移、配置模板或制品摘要变化后，原 PASS 自动失效。

Sandbox PASS evidence 必须包含五份网络实测证据的 SHA256：应用运行网络、应用重启持久化、数据库重启持久化、第三台未授权来源拒绝、数据库侧 REJECT 计数器增长。缺少任一证据时，JWT/API E2E 不得生成 PASS。

## 6. G4 Pilot 准入门禁

- Pilot 必须部署 Sandbox 已验证的同一 GHCR 镜像摘要，禁止重新构建；制品清单、SBOM 和 provenance 必须随审批单流转。
- Pilot 安装器必须校验 Sandbox PASS evidence、审批清单 SHA256、Git SHA、OCI revision、前后端摘要和迁移摘要；禁止 `:pilot` 等可变标签及目标主机本地构建。
- 数据库迁移必须先完成备份、兼容性和回滚评审。
- 发布窗口、影响 tenant、观察指标、停止阈值和回滚负责人必须明确。
- Pilot 仍禁止真实平台连接、高风险 RPA 和资金动作，除非对应专项评审单独批准。
- Pilot 验证失败必须停止流量、回滚制品并保留审计证据。

## 7. Production 与真实平台边界

Sandbox 或 Pilot PASS 均不构成 Production 或真实平台授权。以下事项必须分别评审：

- BigSeller、Shopee、TikTok/TK 等真实平台凭据托管、轮换和网络白名单。
- 真实 webhook、回调域名、签名、限流和灰度策略。
- 自动上架、改价、清仓、停售、归档、补货和真实 RPA。
- 银行、支付、付款、转账、提现和财务自动确认。
- 生产容量、监控、备份、灾备、隐私和合规。

未获得专项批准时，上述能力必须保持 `disabled`，页面不得展示为 `connected`。

## 8. 回滚与停止条件

任一条件触发立即停止晋级或回滚：

- 跨 tenant、越权、敏感数据或凭据泄露。
- 数据迁移不可逆或备份恢复失败。
- 核心健康检查、登录、权限、任务队列或数据库失败。
- 错误率、延迟、资源使用或任务积压超过批准阈值。
- 制品摘要、配置或审核 HEAD 与批准记录不一致。
- 发现真实外部连接或高风险动作被意外启用。

回滚只允许由受控运维流程执行；前端、业务 API 和 RPA 不得提供系统回滚命令入口。

## 9. 审批记录

每次晋级至少记录：申请编号、环境、Git SHA、镜像摘要、迁移版本、测试报告、风险清单、回滚方案、申请人、架构审核人、安全审核人、业务验收人、发布负责人、决定和时间。创建人不得单独批准自己的发布。
