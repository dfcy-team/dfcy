# 阶段2 API 接入与访问安全方案

## 平台接入原则

BigSeller、Shopee、TK/TikTok 的平台调用必须由后端连接器发起，前端不得直接调用第三方平台，RPA 也不得使用平台 API 代替页面任务。阶段2默认使用 Mock 或沙箱；真实接入必须通过专项安全评审。

## 凭据托管与存储

- API Key、API Secret、Token 和刷新凭据禁止明文入库，禁止进入 Git 仓库、日志、截图、payload 或前端构建产物。
- 真实凭据仅可由部署环境的密钥管理能力注入；示例文件只使用 `change-me`、`placeholder`、`demo` 或 `example.com`。
- 如确需在业务库保存可轮换的加密引用，应使用受控密钥进行字段级加密，保存密钥标识、版本、到期时间和审计元数据，且解密只在后端最小权限服务内进行。

## 请求保护

- 使用请求签名、时间戳、nonce 和允许的签名算法，防止重放与篡改。
- 按 tenant、平台、店铺逻辑范围和连接器实施限流、超时、指数退避和重试上限。
- 使用幂等键和同步批次，避免重复写入最终可信业务数据。
- webhook 必须验证签名、时间窗口、来源和幂等事件标识；校验失败不得落库。

## 数据与日志

- 外部数据落库前必须执行字段标准化、类型和枚举校验、tenant 归属校验、重复检测和质量标记。
- API 日志仅保留必要元数据，Authorization、Cookie、Token、Secret、身份证明、银行信息和完整敏感请求体必须脱敏或排除。
- 失败应产生脱敏错误码、告警事件、重试状态和人工处理入口；不得泄露凭据。

## 灰度与沙箱

先以 Mock 验证接口契约，再以批准的沙箱做小范围灰度验证。灰度应可按 tenant、连接器和同步类型关闭，并具备停止、回退和审计能力。未完成接口必须标记为 mock 或 pending。

## 真实平台专项安全评审清单

1. 平台用途、数据分类和最小权限范围。
2. API Key/API Secret 的申请、托管、轮换、撤销和泄露处置。
3. 网络出口、IP 白名单、TLS、webhook 回调与证书策略。
4. 请求签名、限流、幂等、重试、告警、审计和数据保留策略。
5. tenant 隔离、供应商和财务数据访问边界。
6. 灰度、回退、紧急停止和人工接管方案。
7. 安全负责人和业务负责人书面批准。
