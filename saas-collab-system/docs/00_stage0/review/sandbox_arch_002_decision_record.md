# SANDBOX-ARCH-002 架构决策记录

## 1. 背景

当前仅有一台应用虚拟机和一台数据库虚拟机，需要保留给正式运行环境，无法同时维持独立三主机Sandbox。团队按模块分工开发，希望减少重复审核与环境操作。

## 2. 决策

采用“开发人员本地Sandbox + GitHub CI不可变制品 + 精简生产发布门禁”。模块按L1/L2/L3风险分级；无P0/P1时不固定执行多轮复审，只有发现阻断问题才进行定向整改和独立复审。

## 3. 影响

- `SANDBOX-ARCH-001`保留为严格独立环境参考，不再是当前默认交付链路。
- 开发人员承担模块本地验证，CI承担全量集成、制品生成和可追溯证据。
- 现有两台虚拟机只承载正式环境，不运行开发Sandbox。
- 生产只部署受保护main生成的同一镜像digest。
- 缺少独立Staging会增加运行环境差异风险，因此迁移预检、备份、冒烟、观察和回滚不得省略。

## 4. 不可降低项

tenant、权限、data_scope、财务隔离、密钥保护、不可变制品、数据库备份、回滚、高风险人工确认和真实平台专项评审均保持强制。

## 5. 结论

**APPROVED_FOR_DOCUMENTATION_AND_IMPLEMENTATION_PREPARATION**

允许按本决策建设模块化Local Sandbox和精简CI/发布流程。具体本地Compose Profiles、合成数据和自动验证脚本完成并通过独立审核前，不声称开发环境标准化已经落地。
