# 用户界面功能运用设计测试与验收计划

## 1. 测试原则

- 每期测试绑定确定的 commit、环境、配置模板和测试数据版本。
- 结果仅允许 PASS、FAIL、BLOCKED、NOT RUN；未执行项必须说明原因。
- 自动化测试、浏览器验证、API测试和安全检查相互补充，不以截图代替权限测试。
- 所有数据使用 demo/placeholder；不得接入真实平台、银行、支付或真实RPA。

## 2. 通用验收矩阵

| 类别 | 验证内容 | 最低证据 |
|---|---|---|
| 构建 | `npm ci`、`npm test`、`npm run build` | 命令、版本、完整结果、warning记录 |
| 后端 | Django check、migration一致性、专项pytest、全量pytest | 命令、通过数、失败详情 |
| 页面 | 桌面/窄屏、菜单、面包屑、筛选、列表、详情、时间线 | 截图、视口、用例编号 |
| 状态 | loading、empty、error、401、403、404、409、422、partial、offline | Mock或测试API响应及页面证据 |
| 认证 | 登录、刷新、退出、过期、禁用账号、并发刷新 | 网络记录和自动化断言 |
| 权限 | 菜单、按钮、字段、data_scope、流程、导出 | 正向和越权拒绝证据 |
| tenant | 跨租户按ID访问、筛选、聚合和导出 | 403/404、审计和数据库断言 |
| 安全 | 密钥扫描、路径扫描、敏感字段、真实平台连接扫描 | 扫描命令和结果 |
| 部署 | Nginx健康、Backend、Celery、Redis、MySQL和回滚 | `compose ps`、健康响应和日志 |

## 3. 分期专项验收

### UI-P0

- 清点视图、路由、API客户端、Mock和连接状态。
- 清点角色、权限码、data_scope、字段脱敏、导出和审批。
- 对照后端 URL 检查前端调用，不存在接口不得标 connected。
- 运行现有前端测试和构建、后端认证/响应/权限专项测试。

### UI-P1

- 真实登录、Token刷新、`/auth/me`、退出和401回登录。
- 菜单、按钮和字段随权限变化，但直接调用仍由后端拒绝。
- 页面统一状态、重复提交、窄屏和键盘基础测试。
- 工作台数字下钻后与目标列表筛选数量一致。

### UI-P2

- 组织、角色、权限和基础档案的 tenant 隔离。
- 唯一性、启停、引用保护、配置版本、回滚和审计。
- 凭证只显示别名、指纹、引用和轮换时间，不回显明文。

### UI-P3 至 UI-P4

- RPA Agent 只能访问 `/api/rpa/*`，管理前端只访问 internal 管理端点。
- claim并发、心跳超时、最大重试、转人工、撤销和页面变化。
- 审批不可绕过；Webhook 验签、时间戳、去重、重放和断网恢复。
- MySQL活动账号锁约束整改完成前，真实并发RPA用例保持 BLOCKED。

### UI-P5 至 UI-P7

- 业务主链路、待确认、异常闭环、数据质量和人工确认。
- 财务只读、字段脱敏、导出审批、下载时效和审计。
- 双主机端口隔离、加密备份、恢复、回滚和低风险灰度。
- 真实平台接入必须另行执行专项安全评审。
